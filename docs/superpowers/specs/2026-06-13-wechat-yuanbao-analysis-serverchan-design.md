# WeChat-Article-Crawler 元宝解读主链路与异步通知设计

## Goal

将公众号文章解读从“本地模型主链路”调整为“复用 `news` 站现有元宝接口作为默认解读工具”，同时保留服务端内部本地兜底能力，并将 `Server酱` 通知改为“先发文章更新提醒，再异步生成解读”的模式。

本次设计还要求保留前台“重新解读”能力，但前台必须无感知本地链路，不能继续依赖 `127.0.0.1` 一类仅本机可访问的地址，因为当前页面通过 Cloudflare 隧道对外提供访问。

## Scope

本次覆盖：

- 将单篇文章解读默认切换为调用 `news` 站现有元宝解读接口
- 在元宝接口失败、超时或返回空结果时，于服务端内部回退当前本地解读
- 将当前强结构化解读约束改为“总结即可”
- 调整 `Server酱` 通知正文，不再附带解读内容，只保留文章更新信息和统一解读页链接
- 保留并改造前台“重新解读”能力，使其通过公网可访问链路触发
- 为上述流程补充测试与运行验证方案

本次不覆盖：

- 不改造 `news` 站元宝接口内部实现
- 不新增数据库、消息队列或独立任务系统
- 不新增复杂前端框架
- 不将公众号解读页面重构为完整动态站点

## Current Context

当前项目已具备以下基础能力：

- 抓取入口在 `scripts/wechat_article_crawler/wechat_crawler.py`
- 单篇解读逻辑在 `scripts/wechat_article_crawler/article_analysis.py`
- 单篇解读结果会写入 `output/article_analysis/<article_id>.json`
- 静态聚合页为 `output/article_analysis/index.html`
- 当前页面“重新解读”按钮已经存在设计基础，但 JS 直接指向本地 `127.0.0.1:8766/api/reanalyze`
- `Server酱` 当前通知链路会拼出文章信息，并可夹带解读内容

`news` 站已具备可复用的元宝解读接口：

- 主接口为 `POST /api/telegraph/interpret`
- 默认走元宝 worker
- 可复用现有账号 profile 和元宝登录态

当前痛点：

1. 公众号解读仍以本地模型链路为主，不符合“优先用元宝”的目标
2. 当前解读提示词偏强结构化，不适合“总结即可”的诉求
3. `Server酱` 通知若等待解读完成，会拖慢主流程
4. 当前前台重解读若继续使用本地地址，在 Cloudflare 隧道场景下会直接失败

## User Problem

用户明确提出的业务要求有四个：

1. 公众号文章默认用 `news` 站现有元宝接口解读
2. 本地解读只作为服务端内部兜底，前台无感
3. `Server酱` 通知只负责提醒文章更新，并在最后附上统一解读页链接
4. 前台仍要支持按条目点击“重新解读”，且该入口必须能在公网访问场景下工作

## Requirements

### Functional Requirements

1. 单篇文章解读时，系统应优先调用 `news` 站元宝解读接口。
2. 元宝接口失败、超时、返回空内容或非成功结果时，系统应自动回退当前本地解读。
3. 解读提示词应从当前字段化提取改成“总结即可”的轻约束模式。
4. 解读结果无论来自元宝还是本地兜底，都应继续落盘到现有 `output/article_analysis/` 目录，并可被聚合页读取。
5. `Server酱` 通知发送时不应等待解读完成，只触发异步解读任务。
6. `Server酱` 通知正文不应包含解读内容，只保留分类公众号、更新时间、标题和统一解读页链接。
7. 前台“重新解读”按钮必须继续可用，并且应通过公网可访问地址或同源相对路径调用后端接口。
8. 前台重新解读触发后，应走与自动解读相同的“元宝优先，本地兜底”流程。
9. 前台不应暴露本地服务地址、内网地址或本地回退细节。

### Non-Functional Requirements

1. 不能因为元宝接口慢或失败而阻塞主抓取和通知链路。
2. 页面和通知对用户展示的入口应保持统一，避免本地地址泄漏。
3. 新增运行期产物应继续落在 `output/` 或 `logs/` 目录体系。
4. 失败场景下应保留足够日志，但日志不应污染通知正文。

## Options Considered

### Option A: 元宝作为主链路，本地仅服务端兜底

做法：

- 自动解读和前台重解读都先调用 `news` 元宝接口
- 失败时在服务端内部回退当前本地解读
- 通知与前台统一只暴露公网解读页和公网可访问重解读入口

优点：

- 最符合用户“元宝作为默认工具”的目标
- 前台无感本地链路
- 元宝不可用时仍保留最后一层解读能力

缺点：

- 需要补一层结果兼容映射
- 需要处理公网重解读入口改造

### Option B: 本地为主，元宝仅兜底

做法：

- 保留当前本地解读主链路
- 仅在失败时调用 `news` 元宝接口

优点：

- 代码改动较小

缺点：

- 与“元宝作为默认解读工具”的目标相反
- 也无法充分利用现有元宝账号和效果优势

### Option C: 只用元宝，不保留本地兜底

做法：

- 全部切到 `news` 元宝接口
- 删除或绕开当前本地解读链路

优点：

- 调用链最简单

缺点：

- `news` 站或元宝登录态异常时无最后兜底
- 风险偏高

## Recommended Approach

推荐采用 Option A：元宝作为主链路，本地仅服务端兜底。

原因：

- 满足“元宝默认解读工具”的明确要求
- 保留本地解读作为后备，不会因为单点失败导致整条链路失效
- 可以同时兼顾自动解读、通知和前台重新解读三个入口的一致性

## Architecture

整体方案分四层：

1. 抓取与调度层
   - 文章抓取完成后，立即进入“通知”和“异步解读调度”两个分支
   - 通知先发送，不等待解读完成

2. 解读执行层
   - 默认调用 `news` 站元宝接口
   - 元宝失败后，服务端内部调用当前本地解读逻辑

3. 结果标准化层
   - 将元宝自由文本总结和本地兜底结果统一映射为聚合页可消费的结构
   - 保证现有 `article_analysis` 页面不因结果来源不同而破坏显示

4. 对外访问层
   - `Server酱` 只提供统一解读页公网链接
   - 前台“重新解读”按钮通过公网可访问或同源相对路径调用，不再直接写死本地地址

## Data Flow

### 自动抓取流程

1. 抓取文章元数据与正文
2. 立即发送 `Server酱` 通知
3. 在后台触发单篇异步解读任务
4. 解读任务优先调用 `news` 元宝接口
5. 若元宝失败，则服务端内部回退本地解读
6. 结果写入 `output/article_analysis/<article_id>.json`
7. 重建 `output/article_analysis/index.html`

### 前台重新解读流程

1. 用户在聚合页点击“重新解读”
2. 前端调用公网可访问的 `POST /api/reanalyze`，推荐使用相对路径或基于公开 base URL 生成
3. 后端复用同一条“元宝优先，本地兜底”的解读执行逻辑
4. 成功后刷新缓存与聚合页
5. 前端提示成功并刷新页面或更新状态

## Prompt And Result Strategy

### Prompt Change

当前公众号解读提示词要求输出结构化 JSON，包含 `topic`、`core_points`、`audience`、`risks` 等字段。

本次调整为：

- 约束词改成“总结即可”
- 元宝主链路不再强求完整字段化 JSON
- 优先拿到可直接阅读的总结文本

### Result Normalization

由于聚合页当前偏向结构化展示，本次需要增加一层标准化策略：

1. 若元宝返回的是纯文本总结：
   - 直接写入 `summary`
   - 同时可选填充兼容字段，例如将全文摘要映射到 `core_points` 的单项或单独显示区域

2. 若本地兜底仍输出旧结构：
   - 保留旧字段

3. 聚合页渲染时：
   - 优先显示 `summary`
   - 若缺少 `summary`，再回退显示旧字段化内容

这样可以实现“提示词更轻”和“页面不崩”两个目标同时成立。

## API Design

### 元宝解读调用

服务端新增一个统一封装函数，内部请求 `news` 站现有接口，建议请求字段包括：

```json
{
  "title": "文章标题",
  "content": "公众号正文文本或 markdown",
  "time": "发布时间",
  "provider": "auto",
  "mode": "web",
  "speed": "fast"
}
```

封装函数对外只暴露统一结果，不向上层透出元宝接口细节。

### 前台重新解读接口

推荐继续保留 `POST /api/reanalyze` 语义，但改造为可被公网访问：

请求体：

```json
{
  "article_id": "abc123",
  "url": "https://mp.weixin.qq.com/s/xxxx"
}
```

成功响应：

```json
{
  "status": "ok",
  "article_id": "abc123",
  "source": "yuanbao",
  "title": "文章标题"
}
```

兜底成功响应：

```json
{
  "status": "ok",
  "article_id": "abc123",
  "source": "local_fallback",
  "title": "文章标题"
}
```

失败响应：

```json
{
  "status": "error",
  "article_id": "abc123",
  "reason": "reanalyze_failed"
}
```

接口约束：

- 前端使用相对路径 `/api/reanalyze`，或通过公开 base URL 配置生成完整地址
- 禁止在 HTML 中硬编码 `127.0.0.1`、`localhost` 或内网地址
- 服务端内部可继续监听本地端口，但需要通过现有公网域名或同源代理暴露出去

## Notification Design

### Single Article Notification

单篇通知正文调整为：

- 分类公众号
- 更新时间
- 标题
- 解读页链接：`https://wx.coco777.vip/article_analysis`

不再包含：

- AI 解读正文
- 本地重解读地址
- 本地文件路径

### Batch Notification

批量通知与单篇通知保持同一口径：

- 只发文章更新信息
- 最后一行追加统一解读页链接
- 不等待批量解读全部完成

## Frontend Behavior

聚合页的“重新解读”能力保留，但交互调整如下：

1. 点击按钮后，按钮进入处理中状态
2. 通过相对路径或公开 base URL 调用 `POST /api/reanalyze`
3. 后端返回成功后，页面提示“重新解读完成，正在刷新”
4. 页面刷新后读取最新聚合结果
5. 若失败，显示失败提示，但不暴露内部兜底细节

需要特别避免的旧行为：

- 不再把接口地址写死为 `http://127.0.0.1:8766/api/reanalyze`

推荐改法：

- 默认前端调用 `new URL('/api/reanalyze', window.location.origin)`
- 或在页面生成时注入一个公开的 `analysisApiBaseUrl`

## Code Changes

### `scripts/wechat_article_crawler/article_analysis.py`

新增或调整：

- 调整单篇解读提示词，改为“总结即可”
- 新增结果标准化逻辑，兼容元宝纯文本总结与本地结构化结果
- 聚合页渲染优先展示 `summary`
- 前端重解读 JS 从“固定本地地址”改为“相对路径或公开 base URL”

### `scripts/wechat_article_crawler/wechat_crawler.py`

新增或调整：

- 抽出统一的“单篇解读执行函数”
- 在其中优先调用 `news` 站元宝接口
- 失败时回退当前本地解读
- 抓取流程中改为“先通知，后异步解读”
- 重解读接口复用同一套执行逻辑

### Configuration

建议新增以下配置项：

- `NEWS_INTERPRET_BASE_URL`
  - 指向 `news` 站解读接口域名或服务入口

- `WECHAT_ANALYSIS_PUBLIC_BASE_URL`
  - 用于生成前台可访问的重解读 API 和聚合页链接
  - 例如 `https://wx.coco777.vip`

- `WECHAT_ANALYSIS_REANALYZE_PATH`
  - 默认为 `/api/reanalyze`
  - 便于后续通过代理改路径

## Error Handling

需要处理以下失败场景：

1. `news` 元宝接口超时或不可达
   - 自动切换本地兜底
   - 记录日志
   - 不阻塞通知发送

2. 元宝返回空内容
   - 视为失败并触发本地兜底

3. 本地兜底也失败
   - 记录失败状态
   - 保留原有缓存或写入失败标记
   - 前台提示“解读失败，请稍后重试”

4. 前台调用到错误地址
   - 通过配置和相对路径设计避免
   - 测试中覆盖页面生成后的 API URL

5. 聚合页重建失败
   - 记录日志
   - 不影响通知和解读主流程

### Unit Tests

补充到 `tests/test_article_analysis.py`：

- 元宝纯文本结果标准化测试
- 本地结构化结果兼容测试
- 聚合页优先显示 `summary` 的测试
- 前端重解读 API 地址不再包含 `127.0.0.1` 的测试

补充到 `tests/test_wechat_auth_expired.py` 或新增对应测试：

- 元宝接口失败时触发本地兜底的测试
- 本地兜底也失败时返回失败状态的测试

### Notification Tests

补充或新增测试验证：

- 单篇通知正文不包含解读内容
- 批量通知正文不包含解读内容
- 通知末尾固定追加 `https://wx.coco777.vip/article_analysis`

### Runtime Validation

1. 运行单篇抓取
2. 确认通知先发出，且正文只包含公众号分类、时间、标题和解读页链接
3. 确认后台随后生成解读结果
4. 打开公网解读页，确认可查看最新总结
5. 点击“重新解读”
6. 确认前台请求地址不是本地 `127.0.0.1`
7. 确认重新解读成功后页面刷新
8. 人工模拟 `news` 接口失败，确认本地兜底能接管

## Risks

1. `news` 站元宝接口若返回格式波动，标准化层需要更强容错。
2. 当前聚合页若仍依赖旧字段展示，页面模板需要同步做兼容改造。
3. 若公网域名未配置对应代理或转发，前台“重新解读”仍可能打不到后端。
4. 通知改成“先通知后解读”后，用户可能在短时间内打开页面看不到最新结果，需要页面文案接受这一时序。

## Acceptance Criteria

满足以下条件即视为完成：

1. 公众号文章默认走 `news` 站元宝接口解读。
2. 元宝失败时，系统会自动回退当前本地解读，且前台无感知。
3. 解读约束词已从结构化提取改为“总结即可”。
4. `Server酱` 通知正文不再包含解读内容，只保留文章信息和 `https://wx.coco777.vip/article_analysis`。
5. 通知发送不等待解读完成，而是异步触发解读。
6. 前台“重新解读”仍可用，且不再依赖 `127.0.0.1` 或其他本地地址。
7. 聚合页能展示元宝总结结果，并兼容本地兜底结果。
