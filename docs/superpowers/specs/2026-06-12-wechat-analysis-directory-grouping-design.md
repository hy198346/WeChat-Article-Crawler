# WeChat-Article-Crawler 解读页目录按公众号分类设计

## Goal

在现有 `output/article_analysis/index.html` 解读聚合页基础上，优化顶部“公众号目录”区域，使其不再只是平铺列表，而是按仓库内 `公众号名字` 文档定义的分类分组展示，并在每个公众号入口旁明确展示该公众号的最近更新时间。

目标是让用户在浏览大量公众号解读时，先按内容类别定位，再按最近更新时间判断是否值得进入查看。

## Scope

本次覆盖：

- 顶部目录区按 `公众号名字` 文档中的分类组织
- 每个目录项展示公众号名称、篇数、最近更新时间
- 文档中未列出的公众号统一归入 `misc公众号`
- 为目录分类与时间展示补充测试

本次不覆盖：

- 不改正文区单个公众号卡片的折叠逻辑
- 不新增搜索、筛选、排序切换
- 不新增新的分类来源或在线配置
- 不改变“正文区公众号块按最近更新时间倒序”的现有规则

## Current Context

当前聚合页已经具备：

- 顶部目录锚点跳转
- 正文区按公众号分组
- 每个公众号标题旁显示“篇数｜最新时间”
- 正文区整体按最新更新时间倒序排列

当前不足：

- 顶部目录是单层平铺，公众号数量多时不便浏览
- 用户希望目录按内容分类查看，而不是只按时间混排
- 目录项本身没有直接显示最近更新时间，必须跳到正文区才能判断新旧

## Classification Source

分类来源固定使用仓库根目录文件：

- `公众号名字`

当前文档定义的分类顺序为：

1. `daily公众号`
2. `行业公众号`
3. `投研公众号`
4. `网红公众号`
5. `官媒公众号`
6. `misc公众号`

文档中未列出的公众号：

- 不单独新增“未分类”
- 统一自动归入 `misc公众号`

## Requirements

### Functional Requirements

1. 系统应读取 `公众号名字` 文档中的分类定义。
2. 顶部目录应按文档中的分类顺序渲染，而不是按全局单层平铺。
3. 每个分类下应只展示当前聚合页中实际存在的公众号。
4. 每个目录项应显示：
   - 公众号名
   - 该公众号解读篇数
   - 最近更新时间
5. 目录项点击后仍应跳转到对应公众号正文分组锚点。
6. 文档中未列出的公众号应自动归到 `misc公众号`。
7. 若某个分类当前没有任何公众号，目录中不应渲染该分类块。
8. 正文区中每个公众号块的顺序仍保持按最近更新时间倒序。

### Non-Functional Requirements

1. 不引入新的运行依赖。
2. 分类文件解析失败时，页面仍应可生成；可回退为全部归入 `misc公众号` 或单层目录。
3. 页面继续保持纯静态 HTML + 内嵌 CSS。
4. 不改变现有 `output/article_analysis/*.json` 作为数据源的模式。

## Options Considered

### Option A: 继续按时间平铺目录，只补最近更新时间

优点：

- 改动最小
- 不需要读取额外文档

缺点：

- 不能满足“目录要分类”的明确需求

### Option B: 使用 `公众号名字` 文档做静态分类映射

优点：

- 完全贴合用户指定口径
- 分类稳定、可控、容易维护
- 不需要引入模型判断或额外配置系统

缺点：

- 新公众号需要依赖文档维护，未维护前会进入 `misc公众号`

### Option C: 用公众号名称关键词自动猜测分类

优点：

- 不依赖人工文档维护

缺点：

- 分类结果不稳定
- 无法保证与用户已有口径一致

## Recommended Approach

推荐采用 Option B：使用 `公众号名字` 文档做静态分类映射。

原因：

- 用户已经明确要求“用公众号名字文档里的分类”
- 仓库里已有现成文件，不需要再发明新规则
- 与当前静态页面生成方式最匹配

## Design

### Directory Structure

顶部目录区域改为“多组目录块”结构：

1. 分类标题
   - 例如 `daily公众号`
   - 可附带该分类当前公众号数量

2. 分类内公众号列表
   - 每个公众号仍是可点击锚点
   - 目录项文案格式：
     - `公众号名（篇数）｜最新：2026-06-12 15:00:49`

### Data Flow

页面生成时按以下步骤处理：

1. 继续扫描 `output/article_analysis/*.json`
2. 得到已去重、已清洗后的公众号条目列表
3. 读取 `公众号名字` 文档，构建 `公众号名 -> 分类名` 映射
4. 对每个公众号计算：
   - 篇数
   - 最新时间
   - 锚点 ID
5. 将公众号分配到对应分类
6. 按文档顺序渲染顶部目录分类块
7. 正文区仍按最近更新时间倒序输出公众号内容块

## Parsing Rules

`公众号名字` 文档按“分类标题 + 多行公众号名”的结构解析。

解析规则：

- 形如 `daily公众号：` 的行表示一个新分类开始
- 空行跳过
- 非空且非分类标题的行视为公众号名称
- 公众号名称前后空白需要去除
- 若同一公众号在文档中出现多次，以首次出现的分类为准

回退规则：

- 文档不存在、无法读取或解析异常时：
  - 不阻断页面生成
  - 所有公众号统一归到 `misc公众号`

## Rendering Rules

目录分类块渲染规则：

- 分类块只在该分类下有公众号时展示
- 分类内公众号列表按该公众号最新更新时间倒序
- 若最近时间为空，则回退到文件 `mtime` 推导出的时间展示
- 目录项文本必须直接包含最近更新时间，避免用户再跳到正文区确认

正文区渲染规则：

- 不改变当前正文块的排序与内容结构
- 仍保留每个公众号标题旁的 `篇数｜最新：时间`

## Testing

需要补充以下测试：

1. 目录按 `公众号名字` 文档中的分类顺序渲染。
2. 未出现在文档中的公众号自动进入 `misc公众号`。
3. 空分类不渲染。
4. 每个目录项包含最近更新时间文本。
5. 分类内公众号顺序按最近更新时间倒序。
6. 文档缺失或解析异常时，页面仍可生成。

## Risks And Mitigations

风险：

- `公众号名字` 文档格式被手工改坏，可能影响分类结果
- 新公众号未及时加入文档，会被归到 `misc公众号`

缓解：

- 解析逻辑保持宽松，异常时不阻断页面生成
- 通过测试覆盖典型文档格式
- `misc公众号` 作为兜底分类，避免目录项消失

## Acceptance Criteria

满足以下条件可视为本次完成：

- 顶部目录已不再是单层平铺，而是按 `公众号名字` 文档分组
- 每个目录项显示公众号名、篇数和最近更新时间
- 文档未收录公众号进入 `misc公众号`
- 页面锚点跳转仍正常
- 现有正文展示与折叠行为不回归
