# WeChat-Article-Crawler 长摘要前端展示优化设计

## Goal

在不改动现有解读接口、落盘 JSON 结构和重解读链路的前提下，优化 `article_analysis` 聚合页中长摘要的可读性，让包含多行、标题和列表的 `summary` 内容能被清晰展示，而不是压成一整段文本。

本次设计聚焦“只改前端展示”这一最小方案，优先解决用户当前看到的阅读体验问题，同时保持与已上线的“元宝优先 / 本地兜底 / 前台重新解读”能力兼容。

## Scope

本次覆盖：

- 优化 `output/article_analysis/index.html` 中单篇文章 `summary` 的展示方式
- 兼容现有 `summary` 中常见的多行文本结构，如 `## 结论`、`### 利好`、`- 列表项`
- 保持旧数据与新数据都可正常显示，不要求重新生成所有历史 JSON
- 补充与该展示逻辑对应的测试与页面级 smoke 验证

本次不覆盖：

- 不调整单篇解读 prompt 或远端/本地模型输出协议
- 不修改 `summary` JSON 字段结构
- 不引入完整 Markdown 渲染依赖
- 不重构整个 `article_analysis` 页面
- 不变更 `/api/reanalyze` 接口行为

## Current Context

当前页面中的 `summary` 已经可能包含明显的结构化文本，例如：

- `## 结论`
- `## 影响`
- `### 利好`
- `- 列表项`

但聚合页现在将 `summary` 作为普通文本直接拼接进 HTML，浏览器会折叠换行，因此多段内容最终显示成“大段连续文字”，可读性较差。

当前实现还具有两个特点：

1. 现有数据源已经混合存在“短纯文本总结”和“多行结构化总结”两类内容。
2. 页面展示逻辑由 Python 生成 HTML，并非运行时前端框架，因此展示优化更适合在服务端生成 HTML 时完成。

## User Problem

用户当前的直接问题不是“总结内容错误”，而是“总结可阅读性差”。

具体表现为：

- 一段很长的 `summary` 被挤成连续文本
- `##` / `###` 这类层次信息没有被视觉区分
- `-` 开头的条目没有被展示成列表
- 用户需要自己在脑中重新分段，阅读负担高

## Requirements

### Functional Requirements

1. 页面应保留现有 `summary` 内容，不擅自删减或改写其语义。
2. 多行 `summary` 应按照换行拆分显示，而不是压成单段。
3. 以 `## ` 开头的行应显示为较强层级的小标题。
4. 以 `### ` 开头的行应显示为次级标题。
5. 以 `- ` 开头的行应显示为列表项。
6. 普通文本行应显示为独立段落。
7. 单行纯文本 `summary` 应继续保持简洁展示，不引入冗余空白。
8. 历史页面数据与未来新生成数据都应兼容。

### Non-Functional Requirements

1. 不引入第三方 Markdown 渲染依赖，避免扩大改动面与安全风险。
2. 所有展示内容必须经过 HTML 转义，避免将 `summary` 直接作为不可信 HTML 注入页面。
3. 新样式应足够轻量，不影响现有页面结构和“重新解读”按钮交互。
4. 若 `summary` 只是普通短句，展示效果不应明显退化。

## Options Considered

### Option A: 基于换行和前缀的轻量语义渲染

做法：

- 在生成 HTML 时解析 `summary`
- 按行识别 `## `、`### `、`- ` 和普通文本
- 输出安全的标题、列表、段落 HTML

优点：

- 改动小，兼容现有数据
- 无需引入第三方库
- 可读性提升明显
- 安全边界清晰，易控

缺点：

- 只支持当前关心的常见结构，不是完整 Markdown

### Option B: 完整 Markdown 渲染

做法：

- 将 `summary` 作为 Markdown 输入统一渲染

优点：

- 表达能力最强
- 对更多文本格式兼容更广

缺点：

- 需要引入 Markdown 渲染和安全过滤
- 改动面与测试面更大
- 对当前问题来说属于过度设计

### Option C: 仅保留换行

做法：

- 页面只增加 `white-space: pre-wrap` 或等效处理

优点：

- 改动最小

缺点：

- `##` 和 `-` 仍然只是文本
- 阅读体验改善有限

## Recommended Approach

采用 Option A。

原因：

- 它正好覆盖用户反馈的核心痛点
- 能兼容当前已经存在的长摘要数据
- 不会碰远端解读和 JSON 结构，风险最低
- 相比完整 Markdown 渲染，后续维护成本更低

## Design

### Rendering Strategy

新增一个轻量的 `summary` HTML 渲染函数，负责：

- 接收原始 `summary` 字符串
- 统一按换行拆分并清理空行
- 识别标题、次级标题、列表项和普通段落
- 输出经过 HTML 转义后的安全片段

建议规则：

- `## ` 开头：渲染为块级节标题
- `### ` 开头：渲染为次级标题
- 连续 `- ` 行：合并到同一个 `<ul>` 中
- 其他文本：渲染为 `<p>`

如果整个 `summary` 只有一行，且不含明显结构前缀，则继续使用紧凑展示，避免把短文本展示得过于“重”。

### HTML Structure

现有 `summary` 的输出位置保持不变，但内部结构从：

- `总结：纯文本`

改为：

- `总结：`
- 一个内部容器，承载标题、段落、列表

这样可以保持页面整体布局稳定，同时允许长文本出现更清晰的层次。

### Styling

补充轻量样式即可：

- 节标题比正文更醒目，但不要过大
- 列表项缩进适中
- 段落之间保留小间距
- 与页面当前简洁风格保持一致

不引入复杂颜色体系或卡片重构，确保本次改动只解决“可读性”。

### Backward Compatibility

该方案对旧数据天然兼容：

- 旧的单行总结仍能正常显示
- 旧的多行结构化总结会自动获得更好的展示效果
- 未来如果模型重新回到短总结输出，也不需要额外适配

## Error Handling

1. 若 `summary` 为空，保持当前降级逻辑，继续回退到 `topic/core_points` 展示。
2. 若 `summary` 中存在未知格式，按普通段落渲染，不抛异常。
3. 若 `summary` 中出现 HTML 特殊字符，统一转义后再渲染。

## Testing

需要补充三类验证：

1. 离线渲染测试
   - 单行纯文本 summary
   - 包含 `##` / `###` / `-` 的多行 summary
   - 含 HTML 特殊字符的 summary

2. 页面输出测试
   - `index.html` 中应产出结构化的安全 HTML，而不是把多行内容压成一段

3. Smoke 验证
   - 本地打开 `article_analysis` 页面，确认“中国基金报”这类长摘要显示为分段结构
   - 确认“重新解读”按钮仍可点击，前台 JS 行为不受影响

## Risks

- 如果后续模型输出出现更复杂的 Markdown 语法，本方案不会完整支持。
- 如果解析规则写得过于激进，可能会把普通文本误判成列表或标题。

对应控制方式：

- 仅识别非常明确的前缀格式
- 未命中的内容全部按普通段落处理

## Implementation Notes

预计改动集中在：

- `scripts/wechat_article_crawler/article_analysis.py`
- 相关测试文件 `tests/test_article_analysis.py`

实现时应遵循以下边界：

- 不改 API
- 不改 JSON schema
- 不改重解读请求参数
- 不在此次顺手调整 prompt

## Validation Plan

落地前后的验证顺序：

1. 补测试，覆盖轻量语义渲染逻辑
2. 运行相关测试
3. 重新生成页面或使用现有样本验证 HTML 输出
4. 做一次页面级 smoke，确认展示改善且交互未受损
