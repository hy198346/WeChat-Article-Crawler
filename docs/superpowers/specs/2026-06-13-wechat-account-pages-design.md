# WeChat-Article-Crawler 公众号目录页与单账号页面设计

## Goal

将当前“所有公众号内容都堆叠在一个 `article_analysis/index.html` 页面”的展示方式，调整为：

- 目录页只展示公众号目录
- 每个公众号生成一个独立静态子页面
- 单公众号页面默认只展示最近一条解读全文
- 历史文章默认折叠，先只展示时间和标题
- 用户点击某条历史记录后，再展开该条历史解读

本次设计的目标是降低首页信息密度，提高浏览效率，同时保留现有静态页面方案、现有 JSON 数据源与前台“重新解读”能力。

## Scope

本次覆盖：

- 调整 `article_analysis` 的静态页面生成结构
- 保留目录分组能力，但目录页不再直接展示各公众号解读正文
- 为每个公众号生成独立静态子页面
- 调整历史解读展示方式为“默认折叠、按条展开”
- 补充相关测试和静态页 smoke 验证

本次不覆盖：

- 不改动解读生成链路
- 不改动 `summary/topic/core_points` 数据结构
- 不改动 `/api/reanalyze` 接口协议
- 不引入数据库或动态服务端渲染
- 不改动 server 酱通知逻辑

## Current Context

当前 `build_analysis_index_html()` 已完成以下工作：

- 读取 `output/article_analysis/*.json`
- 按公众号聚合、按时间排序
- 首页顶部展示“公众号目录”
- 首页正文中依次输出每个公众号的最新一条解读
- 历史解读通过一个大 `details` 块整体折叠

当前痛点：

1. 首页虽然有目录，但正文仍然把所有公众号的最新解读都展开，页面过长。
2. 一个公众号历史较多时，历史区会一次性挂出多条完整解读，阅读负担大。
3. 用户如果只想看某个公众号，需要先在目录中定位，再回到同一页内浏览，缺少真正独立页面。

## User Problem

用户当前想要的不是继续优化单页排版，而是调整整个信息架构：

- 首页只做“入口页”
- 每个公众号拥有自己的独立页面
- 单公众号页重点展示“最近一条”
- 历史内容默认不打扰，只在用户点击具体历史条目时展开

这本质上是从“单页聚合阅读”切换到“目录 + 单账号详情页”的浏览模型。

## Requirements

### Functional Requirements

1. `article_analysis/index.html` 应只展示公众号目录，不再直接展示各公众号完整解读正文。
2. 每个公众号应生成一个独立静态子页面。
3. 目录项应至少包含：
   - 公众号名
   - 文章数量
   - 最近更新时间
   - 最近标题
4. 点击目录项应跳转到对应公众号页面，而不是同页锚点。
5. 单公众号页面应默认展示最近一条文章的完整解读内容。
6. 单公众号页面的历史文章应默认折叠。
7. 历史文章折叠态应先只显示时间和标题。
8. 点击某条历史文章后，应只展开该条历史解读内容。
9. 最近一条文章仍保留现有“重新解读”按钮能力。
10. 历史条目如已有原文链接，也应保留原文可点击能力。

### Non-Functional Requirements

1. 继续使用静态 HTML 输出，不引入运行时模板引擎或前端框架。
2. 目录页与单公众号页都应保持当前简洁样式风格。
3. 页面生成过程应继续基于现有 `output/article_analysis/*.json` 数据源。
4. 生成路径应稳定、可预测，便于 Cloudflare/静态服务托管。
5. 内容展示必须继续经过 HTML 转义，避免注入风险。

## Options Considered

### Option A: 目录页 + 每个公众号一个静态子页面

做法：

- `index.html` 只作为目录入口
- 每个公众号输出到 `article_analysis/accounts/<slug>.html`
- 历史条目在单公众号页内按条折叠展开

优点：

- 最符合用户诉求
- 首页最清爽
- 单公众号页面 URL 明确，分享和定位更方便

缺点：

- 需要新增页面生成逻辑和 slug/path 管理
- 测试面比单页改造稍大

### Option B: 单页保留，只将目录跳转改为锚点

做法：

- 保持只有一个 `index.html`
- 目录项跳转到页面内对应公众号区域
- 历史改为按条折叠

优点：

- 改动较小

缺点：

- 不符合“每个公众号单独一个网页”
- 首页仍会很长

### Option C: 一个页面 + 查询参数过滤

做法：

- 保持一个 HTML
- 用 `?account=xxx` 决定只展示某一公众号

优点：

- 生成页面数量少

缺点：

- URL 语义不如静态子页面直观
- 静态环境下实现和测试都会更绕

## Recommended Approach

采用 Option A。

原因：

- 它最直接满足“目录页”和“单公众号单独网页”两个关键要求。
- 仍然保持静态页面生成模式，和当前部署方式兼容。
- 后续如果想扩展公众号级 SEO、分享链接、分页等能力，这个结构也更自然。

## Design

### Output Structure

建议输出结构如下：

- `output/article_analysis/index.html`
- `output/article_analysis/accounts/<slug>.html`

其中：

- `index.html` 为公众号目录页
- `accounts/` 下为每个公众号的独立静态页面

`slug` 应来自稳定、可重复的账号标识生成方式，避免因标题变化影响 URL。

### Directory Page

目录页只负责导航，不再承载解读正文。

每个目录项显示：

- 公众号名
- 篇数
- 最近更新时间
- 最近标题

目录页仍可保留现有分类分组（如“行业公众号”“财经媒体”等），但每一项的跳转目标改为对应静态子页面，而不是锚点。

### Account Detail Page

每个公众号页面包含：

- 页面标题：公众号名
- 返回目录页链接
- 最近一条文章的完整解读块
- 历史文章列表

最近一条文章区域继续沿用当前单条卡片展示逻辑，包括：

- 标题
- 时间
- 原文链接
- 重新解读按钮
- AI 解读内容

### History Interaction

历史区不再一次性展开所有历史解读全文，而是调整为：

- 默认只展示“时间 + 标题”
- 每条历史记录单独作为一个可展开项
- 用户点击具体历史项后，才显示该条历史解读

建议使用原生 `details/summary` 结构逐条实现，原因是：

- 静态 HTML 足够实现
- 默认折叠行为天然可用
- 不需要引入额外 JS 状态管理

### Rendering Boundaries

建议将当前单条条目渲染逻辑进一步拆成两个层次：

- 目录项渲染
- 公众号页内“最新条目”与“历史条目”渲染

这样可以避免继续把所有页面输出逻辑塞在同一个大函数里，也便于分别测试：

- 目录页是否只显示目录
- 单公众号页是否只显示本账号内容
- 历史条目是否默认折叠

### Path and Linking

目录页到单公众号页的链接建议使用相对路径：

- `accounts/<slug>.html`

单公众号页返回目录建议使用：

- `../index.html`

这样无论本地静态服务还是现有线上静态托管，都更稳定。

## Error Handling

1. 若某公众号没有有效文章，跳过生成其详情页。
2. 若标题或时间缺失，继续沿用当前兜底文案。
3. 若某条历史解读没有 `summary`，则回退展示现有 `topic/core_points` 结构。
4. 若 slug 发生冲突，应使用稳定去冲突策略，避免页面覆盖。

## Testing

需要补充以下测试：

1. 目录页测试
   - 只出现目录项
   - 不出现完整解读正文
   - 目录链接指向 `accounts/<slug>.html`

2. 单公众号页测试
   - 页面只出现该公众号内容
   - 最近一条完整解读默认展示
   - 返回目录链接存在

3. 历史折叠测试
   - 历史条目默认使用逐条折叠结构
   - 折叠标题只包含时间和标题
   - 展开内容中才出现对应历史解读

4. smoke 验证
   - 重新生成静态页
   - 检查 `index.html` 和至少一个 `accounts/<slug>.html`
   - 确认链接、折叠结构、最新条目展示都符合预期

## Risks

- 如果 slug 生成规则不稳定，后续页面链接可能变化。
- 如果目录页和单公众号页共用太多分支逻辑，函数会继续膨胀。
- 如果历史条目仍复用当前完整条目渲染函数，可能会把“重新解读按钮”和大段正文错误带入折叠标题。

对应控制方式：

- 采用稳定 slug 规则
- 明确拆分目录页渲染、详情页渲染、历史条目渲染
- 为“历史折叠标题”和“历史展开正文”分别提供独立渲染函数

## Implementation Notes

- `scripts/wechat_article_crawler/article_analysis.py`
- `tests/test_article_analysis.py`

如果当前 `build_analysis_index_html()` 继续增长过快，可以在本次任务中适度拆出小型 helper，但仅限服务于本次目录页 / 单页生成目标，不做无关重构。

推荐验证顺序：

5. 本地检查：
   - 目录页是否只剩公众号目录
   - 单公众号页是否默认展示最近一条解读
   - 历史条目是否默认折叠并按条展开
